# 🧠 Mental Management Center (MMC) — Гуманітарна вебплатформа взаємодії

**Mental Management Center (MMC)** — це архітектурно зріла, криптографічно захищена вебплатформа, розроблена для забезпечення повноцінної асинхронної та синхронної взаємодії між незалежним фахівцем (гуманітарного профілю) та його клієнтами. Проєкт орієнтований на безкомпромісне дотримання приватності, захист конфіденційних даних рефлексії та стабільну роботу в реальному часі.

Платформа є некомерційним, гуманітарним продуктом, не пов'язана з медичними протоколами та базується суворо на етичних стандартах конфіденційності гуманітарної практики. Застосунок повністю розгорнуто та адмініструється на власному автономному сервері під керуванням OS Linux (Lubuntu) із забезпеченням апаратної відмовостійкості (Fault Tolerance інфраструктура на базі ДБЖ та генератора).

---

## 🤖 AI-Human Co-Creation & Чесна розробка

Проєкт MMC v1.0 є продуктом **гібридної розробки (AI-Assisted Software Engineering)**. Автор проєкту виступав у ролі продуктового архітектора системи, автора бізнес-логіки та унікальних модулів безпеки, а реалізація, точкове закриття багів та оптимізація коду здійснювалися у щільній синергічній взаємодії з LLM-моделями (Google Gemini). Такий підхід демонструє інженерне вміння автора ефективно керувати сучасними інструментами штучного інтелекту для швидкого прототипування, розробки та стабілізації комплексних Enterprise-систем.

## 🛠️ 1. Повний технологічний стек

* **Backend Core:** Java 17, Spring Boot 3.2 (Web, Security, Data JPA, Validation, WebSocket, Mail)
* **Автентифікація & Протоколи безпеки:** Spring Security, **OAuth 2.0 / OIDC (OpenID Connect)** для безшовної та безпечної авторизації користувачів (включаючи інтеграцію з Google Login).
* **База даних & ORM:** PostgreSQL (Production), Hibernate ORM, Liquibase (інтегровано на рівні конфігурації під майбутню міграцію)
* **Прикладна криптографія:** JCA (Java Cryptography Architecture), шифрування AES-256-GCM, стиснення GZIP
* **Мережеві протоколи & Real-Time:** WebSockets (STOMP / SockJS), WebRTC (Jitsi Meet API + RSA-256 JWT)
* **Frontend:** Thymeleaf (SSR), Vanilla JavaScript (ES6+, асинхронні AJAX-модулі)

## 🏗️ 2. Архітектурні модулі та інженерні рішення

### 🔒 Дворівневий захист бінарних даних (Binary Header Stripping & Crypto)
* **Криптографія + Компресія:** Текстові записи щоденників стискаються через GZIP (для оптимізації дискового простору та зменшення `byte[]` в БД) і шифруються «на льоту» за алгоритмом AES-256-GCM.
* **Деструктуризація файлів (Header Stripping):** При завантаженні будь-якого медіафайлу система «відриває» його перші 4 КБ (заголовок та метадані файлової структури) і зберігає цей фрагмент ізольовано всередині реляційної БД. Основне «тіло» файлу зберігається на жорсткому диску сервера.
* **Результат:** Файли на диску сервера перетворюються на мертвий бінарний шум. Їх неможливо прочитати чи дешифрувати навіть у разі фізичного викрадення дисків. Збирання та дешифрування файлу відбувається виключно в пам'яті застосунку «на льоту» під час легітимного запиту користувача.

### 📡 Мережевий телекомунікаційний хаб та Real-Time
* **Повнодуплексний чат спільноти:** Побудований на базі Spring WebSockets та протоколу STOMP. Навантаження на сервер оптимізовано за допомогою клієнтського кешування та покрокового витягування повідомлень порціями через `Spring Data Slice` (нескінченний скрол).
* **Захищений відеозв'язок (WebRTC):** Інтегровано сервіс Jitsi Meet. Безпека кімнат реалізована через серверне генерування тимчасових JWT-токенів, підписаних асиметричним алгоритмом RSA-256.

### 🗄️ Оптимізація доступу та Рольова модель на бітових масках
* **Ліквідація N+1 Queries:** Повністю вирішено проблему надлишкових SQL-запитів до PostgreSQL при завантаженні складних ієрархічних структур (деревоподібні коментарі) за допомогою точкових JPQL-запитів, механізмів `FETCH JOIN` та декларативних графів сутностей (`@EntityGraph`).
* **Мультирольова Композиція (Bitwise RBAC):** Права 5 рівнів доступу закодовані через степені двійки в одному цілочисельному полі `rolesMask` (кастомний Enum `RoleBit`). Це дозволяє одному користувачу володіти кількома ролями одночасно (наприклад, ADMIN + CLIENT) та готує архітектурний фундамент під масштабування платформи.
* **Контроль доступу до контенту:** Неавторизований гість має доступ лише до прев'ю-версії статей. Повний текст публікацій та медіаматеріали відкриваються суворо після автентифікації (через стандартну форму або OAuth 2.0 провайдерів).

### ⚖️ Гнучка дворівнева деактивація даних
* Платформа надає користувачеві інструмент «Усвідомленого бар'єру», що пропонує йому самостійно видалити свої коментарі та чати перед деактивацією. Для екстрених чи кризових ситуацій передбачено модуль «Бардачок», де Адміністратор має повне право видалити профіль користувача **в один клік**. При цьому `UserService` за одну атомарну транзакцію (`@Transactional`) каскадно очищає всі пов'язані за UUID таблиці.

### 📐 Математична дизайн-система та колірний двигун
* **Повна відмова від пікселів (`px`):** Верстка побудована на відносних одиницях (`rem`, `vh`), квантовій сітці та динамічних обчисленнях `calc()`.
* **Градієнтний спектральний колірний двигун:** Користувач може гнучко налаштувати індивідуальну схему всього інтерфейсу через градієнтний спектр у своєму профілі. Система динамічно вираховує HSL-токени на основі координат кліку та адаптує під них денний чи нічний режими сайту без дублювання CSS-коду.

## 🗺️ 3. Пріоритетний вектор технічного розвитку (Roadmap)

Проєкт орієнтований на поступове підвищення відмовостійкості, безпеки інфраструктури та оптимізацію архітектури:

1. **Міграція на суворе версіонування БД (Liquibase):** Перехід від автоматичного оновлення схем Hibernate (`ddl-auto: update`) до контрольованого управління структурою PostgreSQL через файли міграцій. Це закладається як базовий крок для підвищення безпеки та збереження даних у production-середовищі.
2. **Розділення архітектури (Decoupled Architecture):** Поступова трансформація моноліту — перетворення Spring Boot на чистий Stateless REST API із потенційним перенесенням клієнтського інтерфейсу на Angular.
3. **Оптимізація доступу (PWA):** Написання додаткових рест-контролерів та інтеграція Service Workers для можливості встановлення застосунку як окремого ярлика на робочий стіл користувача.

## 💻 4. Системне адміністрування (Bare-Metal Production)

Проєкт не залежить від хмарних провайдерів та розгорнуту на фізичному обладнанні:
* **SSH-Адміністрування:** Моніторинг процесів БД PostgreSQL та аналіз логів здійснюється виключно через інтерфейс термінала по захищеному протоколу SSH.
* **Fault Tolerance:** Локальна інфраструктура резервного живлення гарантує збереження транзакцій у базі даних при перебоях в електромережі.
